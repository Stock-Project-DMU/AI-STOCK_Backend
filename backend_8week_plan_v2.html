백엔드 개발 계획 — 전원 동시 시작

A·B·C 세 명 모두 1주차부터 실제 코드를 작성합니다. 대기 주차를 없애기 위해 "서로의 결과물이 필요 없는 독립 작업"을 각자에게 먼저 배분하고, 뒤로 갈수록 자연스럽게 서로의 결과물을 이어받는 순서로 재구성했어요.

무엇이 바뀌었나

이전 계획은 project-init 하나가 끝나야 다음 사람이 시작할 수 있다고 가정해서 B·C가 1~2주차를 통째로 대기했어요. 하지만 실제로는 project-init 이후 db-entity · jwt-common · redis-config · common-response · websocket-config 는 서로를 필요로 하지 않는 완전히 독립된 작업이에요. 이 5개를 1주차에 동시에 나눠 배정해서 대기 시간을 없앴습니다.

01 — 독립 작업부터 먼저
project-init만 끝나면 db-entity·jwt-common·redis-config·common-response·websocket-config는 서로 필요 없이 동시에 시작 가능합니다.
02 — 이어받는 작업은 순서대로
auth-login처럼 여러 결과물이 필요한 기능은, 필요한 것들이 이미 앞주에 끝나 있도록 배치해 대기 없이 바로 시작합니다.
03 — 6주 집중 개발 + 2주 통합
21개 브랜치를 6주 안에 나눠 끝내고, 남은 2주는 세 명 모두 통합 테스트·버그 수정에 집중합니다.
쉬움무리 없이 진행 가능 보통개념 이해 + 구현 병행 어려움설계 경험이 필요한 구간

운영 원칙 정리

  • project-init은 build.gradle(A)·application.yml(B)·gitignore+CI(C) 3파일로 쪼개 첫날 동시에 끝냄
  • 1주차부터 세 명 모두 실제 기능 코드 작성 — 이전 계획의 대기 주차를 전부 제거
  • redis-service는 auth용(2개)·market용(3개)으로 나눠 각자 필요한 시점에 스스로 준비
  • 같은 사람이 순서대로 하는 작업(auth 3종, order 2종 등)은 파일이 겹쳐도 충돌 없음
  • 다른 사람의 결과물이 필요한 작업은 필요한 것들이 이미 전주까지 끝나도록 배치해 대기 없이 시작
  • 6주 안에 21개 브랜치 모두 완료, 남은 2주는 전원 통합 테스트·문서화에 집중
  • 브랜치 시작 전 항상 dev pull 후 CLAUDE.md·NAMING.md·schema.sql 최신본 확인
  • 새 클래스·메서드·필드는 코드 작성 전 NAMING.md에 먼저 등록
